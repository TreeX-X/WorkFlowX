# Module 06: Task Coordination

> **Trigger**: Module 05 完成后，持续运行直到所有任务完成或用户中断

## Team Lead 职责

orchestratorX 作为 Team Lead，负责：

1. **任务调度**：监控任务状态，触发依赖解锁
2. **文件冲突检查**：任务分配前检查文件重叠
3. **需求变更处理**：接收用户输入，更新 Hybrid Tree
4. **进度监控**：报告进度，处理异常

**注意**：队友只负责认领任务、完成任务、报告结果。调度逻辑由 Team Lead 统一管理。

## 核心调度循环

```
while (还有未完成的任务) {
  1. 检查已完成的任务
  2. 解锁依赖任务（已完成任务的下游任务）
  3. 文件冲突检查（ready 任务与 in_progress 任务）
  4. 分配任务给空闲队友
  5. 处理用户输入（如果有）
  6. 监控进度
  7. 等待事件
}
```

## 依赖解锁

**触发条件**：任务标记为 completed 时

**解锁逻辑**：
1. 找到依赖此任务的其他任务（blockedBy 字段包含此任务 ID）
2. 检查这些任务的所有依赖是否都已完成
3. 如果是，将任务状态从 `blocked` 改为 `ready`

## 文件冲突检查

**检查时机**：任务分配前

**检查逻辑**：
1. 读取目标任务的文件索引（Child Section 8.1）
2. 读取所有 in_progress 任务的文件索引
3. 如果有文件重叠，任务状态设为 `wait_blocked`
4. 当冲突任务完成后，重新检查并解锁

**冲突判定**：
- 精确匹配：同一文件被多个任务修改
- 不检查目录级别（同目录不同文件不冲突）

## 任务分配

**分配规则**：
1. 优先分配给空闲的 coder-teammate
2. evaluator-teammate 等待 coder 完成后再分配
3. 按依赖数量排序（依赖少的优先）

**分配流程**：
1. 找到所有 `ready` 状态的任务
2. 检查文件冲突
3. 分配给空闲队友
4. 通知队友认领

## 需求变更处理

**处理流程**：遵循主 SKILL.md 的 Requirement Change Handling 章节

**Agent Teams 特殊处理**：
- 变更直接更新 Hybrid Tree
- 队友在下次认领任务时会看到最新文档
- 如果变更影响 in_progress 任务：
  - 非紧急：更新文档，等待完成后再处理
  - 紧急：中断队友，重新分配

## 迭代轮次控制

**控制逻辑**：
- 每个任务有迭代计数器
- 评估结果为 `needs_fix` 时，计数器 +1
- 计数器达到上限（`-N` 参数）时，任务标记为 `failed`
- 报告给用户，等待进一步指示

## 进度监控

**监控内容**：
- 总任务数、已完成、进行中、待分配、阻塞
- 超时检测（可选）

**输出格式**：
```
📊 进度: {completed}/{total} 完成 | {inProgress} 进行中 | {ready} 待分配 | {blocked} 阻塞
```

## 完成处理

**完成条件**：所有任务状态为 `completed` 或 `failed`

**完成动作**：
1. 汇总结果
2. 输出最终报告
3. 清理团队（使用 Agent Teams 的 cleanUpTeam 命令）

## 输出

调度循环结束后，输出：

```
✅ 并行工作流完成

📊 最终统计
   总任务数: {total}
   成功完成: {success}
   失败: {failure}

📁 Hybrid Tree 已更新
   位置: .hybrid/{feature-name}/
```
